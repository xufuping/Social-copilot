# 当前最核心的产品需求

后续每个版本都必须先回答：**这个版本最核心要解决的产品问题是什么？**

当前版本的核心产品目标如下：

1. **让用户在无法直接获取聊天上下文的前提下，仍能低成本拿到可直接发送的多条候选回复。**
   - 候选条数由用户在 **3～10 条** 范围内自选（**默认 3 条**，最少 3 条）；每次发送时在请求侧明确告知模型「提供 n 条回复消息」。
   - 当前主路径是：选择联系人 -> 粘贴对方消息 -> 补充本次交流预期 -> 向 AI 请求建议。
   - 交互仍然坚持“手动触发获取建议”，而不是恢复 Ghost Text 实时自动渲染。
   - 当前版本不再把“自动读取聊天窗口”作为近期成立前提，先把手动输入路径做顺滑。

2. **让 AI 的建议明确围绕“聊天联系人”而不是一次性 Prompt 展开。**
   - 左侧联系人列表由用户手动创建和维护；**新建联系人**通过顶部图标入口打开弹窗，采集**昵称**与**属性定义**（原「关系 / 身份」类信息如「普通朋友」一并写入属性定义，不再单独字段强调）。
   - 每个联系人都沉淀名称、属性定义、蒸馏 `skill` 等信息，后续可升级为 RAG 记忆。
   - AI 输出必须持续受联系人画像约束。

3. **让桌面端成为稳定可用的“联系人 + AI 聊天工作台”。**
   - 左侧是联系人列表，右侧是 AI 聊天框。
   - AI 聊天框**不再**使用独立顶栏：联系人昵称、属性定义摘要、当前联系人的蒸馏 **Skill** 与聊天气泡预览整合在同一主体区域内（原「聊天模拟」标题位由该信息条替换）；下方为 AI 候选回复区、聊天气泡预览双栏、两段式输入区与底部操作区。
   - 即使只使用 Mock、或真实模型因配置/网络暂时不可用、或富能力尚未开放，应用也要保持清晰的空态、失败态与可重试体验。

# 当前技术架构与边界

后续每个版本都必须明确：**当前版本建立在什么已落地架构之上，以及哪些技术边界已经被锁定。**

## 当前已落地的技术基线

- **桌面外壳**：Tauri v2 + Rust
- **前端视图层**：Next.js 16（App Router）+ React 19 + TypeScript
- **样式与组件**：TailwindCSS v4 + Shadcn UI + lucide-react
- **包管理**：pnpm
- **运行模式**：Next.js `output: 'export'` 静态导出，由 Tauri 加载 `out/`
- **窗口形态**：无边框、始终置顶、侧边栏尺寸、自绘拖拽条与窗口控制按钮
- **当前数据来源**：联系人等仍为应用内预置/会话态 Mock 数据，尚未做本地文件持久化（见阶段三）
- **当前 AI 状态**：
  - **Mock 路径**：在前端用 `generateMockSuggestions` 按请求条数 **n（3～10）** 生成建议；可在任意环境用于 UI 自测。**Mock 返回的 `text` 仅呈现可发送的自然中文对白**，不注入 `[Mock · 意图=…]`、`已知画像` 等调试或元信息字符串（画像约束仍在请求侧参与逻辑，但不污染用户可见文案）。
  - **真实模型路径**：通过 Tauri Command `generate_ai_suggestions` 调用，Rust 使用 `reqwest` 向 OpenAI-compatible `chat/completions` 发 HTTP 请求；`baseUrl`、`apiKey`、模型名经 `get_ai_config` / `save_ai_config` 读写应用配置目录中的 `ai-provider.json`，不进入前端 bundle。
  - **运行限制**：`pnpm dev` 纯浏览器（非 Tauri）下无法 `invoke` 真实链路，会提示需桌面端；此时应使用 Mock 模型。

## 当前已明确锁定的架构决策

以下技术路线已经明确锁定，并应视为后续版本继续演进时的默认前提：

1. **AI 调用不再采用 Next.js API Routes / Server Actions。**
   - 原因：当前项目已经选择 Next.js 静态导出模式，这条路与运行方式不兼容。

2. **AI 调用统一采用 Tauri Rust Command 中转。**
   - 前端通过 `invoke(...)` 请求 Rust Command。
   - Rust 负责模型请求、密钥管理、代理兼容、错误处理。
   - `contact.skill`、蒸馏、Prompt Builder、Provider Adapter 等中间层能力优先放在 Rust Command 管线中。

3. **网络代理兼容优先在 Rust 网络层解决。**
   - 后续模型请求建议走 Rust `reqwest` / 可控 HTTP 客户端。
   - 目标是兼容本地代理环境（如 Surge）并保留后续插入中间件的能力。

4. **当前主输入链路采用“用户手动复制粘贴”，而不是系统探针 / OCR / 自动读取其他应用。**
   - 前端负责承载结构化输入与粘贴后的上下文整理。
   - 后端 AI 管线围绕联系人信息、对方消息、用户预期组装请求。

5. **联系人是核心组织单位。**
   - 会话记录、属性定义、蒸馏 `skill`、后续 RAG 记忆都优先挂在联系人维度下。
   - 右侧 AI 工作台始终围绕当前选中的联系人展开。

6. **输入能力按“两段式工作流”设计。**
   - 联系人消息粘贴输入框。
   - 用户预期输入框，合并话题背景、对方状态判断、回复目标、语气风格与禁忌点。
   - 不再单独提供 `topic` 字段或话题描述输入框。
   - 为后续链接聊天应用、引用图片 / 文件、语音输入预留扩展位。

## 当前明确不采用的路线

1. **不回退到 Python / PyQt6 / LangChain-Python 技术栈。**
2. **不采用 Next.js Server Actions / API Routes / ISR 作为桌面端 AI 主链路。**
3. **不采用浏览器端直持 API Key 的直连方案作为默认架构。**
4. **当前版本不以系统探针、OCR、剪贴板监听、自动注入聊天应用作为 MVP 主链路。**

# 当前未完成项与未来规划

## 阶段二：联系人工作台与手动上下文输入闭环

**实现状态（与当前仓库一致）**：主界面结构、每联系人独立工作台、Mock 与 Tauri 真实模型两条链路均已落地；**当前版本规划重心已移至阶段三**（本地持久化与画像正式化），本节的条目继续作为**需求规格与回归验收口径**保留。

**目标**：在无法自动读取外站聊天的前提下，把“联系人选择 / 创建 → 粘贴对方消息 → 补充预期 → 生成 **n（3～10）** 条可发送建议”的闭环定稿并可维护。

### 2.1 左侧联系人列表

1. **列表顶部工具行只占一行**：左侧为列表标题或搜索等（与实现一致），**右侧为「新建联系人」图标按钮**（不占两行列表项高度）。
2. **新建联系人图标**：
   - **Tooltip 标题**：`新建联系人`
   - **Tooltip 描述**：`每个新联系人会独享一个聊天对话上下文`
   - 点击后打开弹窗：输入**昵称**、**属性定义**（凡原「关系 / 身份」表述如「普通朋友」「向上管理」等均作为属性定义的一部分写入，不单独必填字段）。
3. 支持从列表中选择当前聊天联系人，并在右侧同步切换工作台。
4. 列表项仍采用紧凑**两行**卡片（仅针对列表中的联系人项，不含顶部新建行）：
   - 第一行：联系人名称 + 最近更新时间
   - 第二行：**属性定义**摘要（可含原关系语义），单行截断
   - 不展示多行摘要，确保列表可容纳足够多联系人
5. **候选回复条数配置**：在 AI 候选回复区**上方**提供下拉框，可选 **3～10**（默认 **3**）。该数值在每次发送时写入 AI 请求约束（见 2.4），用于提示模型输出对应条数。
6. 为后续扩展预留：联系人状态、最近会话摘要（hover 或展开时展示）等。

### 2.2 右侧 AI 工作台主结构

1. **取消独立顶部横条（原一行联系人 Header）**：不再在双栏上方单独占用一整条顶栏。
2. 中间主体区域为左右双栏：
   - **左侧：聊天模拟 / 预览区**
     - **原「聊天模拟」标题位置**改为**联系人信息条**：从左到右依次为**联系人昵称**、**属性定义**（摘要；**垂直居中**；鼠标悬停属性区域时 `title` 展示**完整属性定义**及可用的 `summary` 等补充）、最右侧为**蒸馏 Skill**（`title` 展示标签/特征/notes 全文）。无有效 skill 内容且非错误态时展示 **「待上传蒸馏 skill」**；`ContactSkill.skillFileError === true` 时展示 **「skill 信息错误」**。
     - 其下为聊天气泡：联系人消息 + 当前选中候选回复预览。**当右侧已存在候选回复列表（发送成功后的状态）时**，即使「联系人发送的内容」已按规则清空，**也不再展示**「粘贴联系人消息后，将在这里预览对话上下文。」类对方空态占位。
   - **右侧：AI 候选回复区**
     - 顶部：**回复条数**下拉（3～10，默认 3）与 **「标题 / 极简」** 控件占用的垂直空间保持**紧凑**（小字号、矮控件），避免挤压消息列表。
     - 列表区：展示 n 条候选回复、加载态、错误态、空态；列表滚动区保留**左右与顶部内边距**，选中态使用 **ring-inset**，避免卡片左边框与首条上边框被裁切。
     - 每条建议卡片**不**展示「本次交流预期」「已知画像」等元信息区块，**仅突出可发送文案**；正文较常规 UI **再小两个字号**（如 `10px` 档），**单行时不保留多余空白行**（正文 `textarea` **高度随内容**自适应）。**标题**模式下卡片紧凑，`style` 作短标题；用户可在正文区直接编辑；支持**一键复制**。**极简**模式下**完全不展示** `style` 类标签文案，复制入口置于**正文行最左侧**。
3. 每个联系人拥有独立的工作台状态（草稿、候选回复、错误信息、回复条数偏好、**style 展示模式「标题 / 极简」**等），切换联系人时恢复之前的内容。
4. 必须提供空态、加载态、失败态。

### 2.3 两段式输入区与底部操作区

1. **版式**：底部为**左右两列栅格**——**左列**「联系人发送的内容」、**右列**「我对本次交流的预期」（提示词 chip 在右列输入框下方）；窄屏可上下堆叠。
2. 第一输入框：联系人发送的内容。
   - 当前主要承载用户复制粘贴的文本消息。
   - **标题置于该列上方**；**可见输入区域默认约三行高度**（可随内容增高，设合理上限）。
   - 右下角预留两个入口：链接、引用（阶段四前可禁用）。
   - 后续可扩展为图片、表情包、文件等内容引用。
3. 第二输入框：我对本次交流的预期。
   - 合并原“本次对话话题描述”与“交流预期”。
   - **标题行**含 **Close** 图标，一键清空预期全文。
   - **默认约三行高度**。
   - 用户可自行输入话题背景、对方状态判断、回复目标、语气风格、禁忌点等。
   - 不再单独维护 `topic` 字段。
   - 提供可点击提示词 chip，至少包括：暧昧、真诚、幽默、严肃、向上管理、普通朋友、委婉拒绝、安抚对方情绪、希望对方合作购买（**「普通朋友」等作为预期风格 chip，与联系人属性定义中的表述可同时存在，语义不同**：前者修饰本次回复意图，后者描述联系人长期画像）。
4. **发送成功后的草稿行为**：每次成功发起一次生成请求后，**清空「联系人发送的内容」**（`incomingMessage`）；**保留「我对本次交流的预期」**（`expectation`），便于连续微调。
5. **`style` 展示配置**：在**右侧**候选回复区提供 **Switch**（或分段控件），在 **「标题」** 与 **「极简」** 间切换（见 §2.2）。**默认「标题」**；该配置可按联系人记忆。
6. **操作区（模型与发送）**：置于**右列**「我对本次交流的预期」**下方**，**右对齐**；**自左向右**顺序为：
   - **设置图标**（打开 Provider / API 配置弹窗，等同原「配置模型」文案入口）
   - **模型下拉选择**（Mock / OpenAI-compatible 等），与设置图标**同一行紧邻**
   - **语音**（未来功能，可禁用）
   - **发送**
   - **不在**全局底栏单独展示「请先粘贴联系人发送的内容」等整行提示；未满足发送条件时以**发送按钮禁用 + `title` 提示**等方式弱提示即可。

### 2.4 与 AI 中转层的对接方式（以当前实现为准）

1. 前端在 **Mock 模式** 下不经过 Rust，直接基于同一套 `AiSuggestionRequest` 生成假数据；在 **OpenAI-compatible 模式** 下调用 Tauri Command `generate_ai_suggestions`。
2. **模型配置**（仅真实链路需要）：`save_ai_config` 写入；`get_ai_config` 读取；未配置时真实链路在命令层返回错误字符串，由右侧工作台展示，输入草稿不丢失。
3. **请求体（`AiSuggestionRequest`，TS/Rust 字段 camelCase 序列化）** 至少包含：
   - `model`：如 `mock`、OpenAI-compatible 时由 UI 与配置共同决定
   - `suggestionCount` 或与实现对齐的等价字段：**整数，范围 3～10**，与 UI 下拉一致；组装 Prompt 时须包含对用户可见约束的语义，例如明确写出 **「请提供 n 条回复消息」**（n 为该字段值）。
   - `contact`：当前选中的 `WorkspaceContact` 画像（属性定义已包含原「关系 / 身份」类信息；若类型中仍暂存 `relation` 等字段，应视为属性定义的镜像或待迁移字段，以 UI 单一「属性定义」为准）
   - `draft`：`incomingMessage`、用户「交流预期」`expectation`（无独立 `topic` 字段；提示词 chip 以追加文本方式写入 `expectation`）
   - `intent`：由 `expectation` 派生的标签/描述（供 Prompt 与后续演进）
   - `recentMessages`：至少包含对方粘贴内容的一条 `Message` 列表
   - `workspaceContext`：如 `source: "manual-paste"` 与 `generatedAt` 等元数据
4. **成功响应**：`Vec<Suggestion>`，长度应与请求的 n 条一致或尽力贴近（Mock 层按 n 生成）；单条为 `{ id, style, text }`。其中 `style` 在解析 OpenAI 响应时由层序占位（如不同候选的区分标签）填充，不依赖多字段 `styleTags`。界面上是否展示 `style` 为标题区由用户的 **「标题 / 极简」** Switch 决定（见 §2.3.4），**默认「标题」**；**极简**模式下**不展示** `style` 标签，以 `text` 为主。
5. **失败形态**：Tauri 命令返回 `Err(String)`，前端将字符串写入当前联系人工作台的 `error`；不单独封装 `debug_info` / 顶层 `error` 对象响应体（若后续要可观测性再扩展）。
6. 选择 Mock 时无需 API Key；选择 OpenAI-compatible 时须在桌面端完成一次有效配置，否则发请求会失败并提示，界面保留用户输入。

### 2.5 阶段二交付标准

- 用户可以手动创建联系人并切换当前联系人（创建入口允许为轻量弹窗/表单，以当前实现为准）。
- 切换联系人时，草稿、候选回复、错误信息与「当前预览哪一条候选」独立保留。
- 用户可以完成“粘贴对方消息 +（可选）补充预期 + 发送”的基本流程；**发送的硬性前置条件**为已选联系人与非空的「联系人发送的内容」；交流预期为强烈建议、非硬性必填（与 `canSend` 逻辑一致）。
- 右侧工作台完整体现最新 UI 结构（**无独立顶栏**；**左**预览列、**右**候选列；预览顶条**垂直居中**，属性 **hover** 与 Skill 态见 §2.2；**已有候选列表时不展示**对方粘贴引导空态；候选区含条数与 **「标题 / 极简」**；输入区**左右分栏**，**设置图标 + 模型 + 语音 + 发送**在**预期列下方**右对齐，**底栏下无**「请先粘贴…」整行文案）。
- 聊天模拟区可预览联系人消息与当前选中候选回复；候选卡片可切换预览；卡片以文案为主，可点击编辑并复制。
- 在 Mock 或已配置完成的 OpenAI-compatible 下，发送后可获得 **n 条**（3～10）候选回复；失败时得到可读错误并允许重试；**成功发送生成后清空「联系人发送的内容」**。

## 阶段三：`contact.skill` 正式化与本地持久化

**目标**：让联系人画像从展示字段变成可积累、可复用的本地长期资产。

### 3.1 联系人画像结构正式化

1. 将联系人数据从前端临时类型升级为跨端共享的数据契约。
2. 至少包含以下字段：
   - `name`
   - `attribute_definition`
   - `manual_tags`
   - `distilled_skill`
   - `notes`
   - `updated_at`
   - （UI 契约）可选 `skill_file_error` / `skillFileError`：蒸馏 skill 文件读取失败时顶栏展示「skill 信息错误」。
3. 明确哪些字段由用户维护，哪些字段由蒸馏器自动生成。

### 3.2 本地存储能力

1. 在应用专属目录下读写联系人数据。
2. 建议按联系人维度存储，例如：
   - `[contact_name].json`
3. 需要处理：
   - 非法文件名
   - 重名联系人
   - Schema 版本升级

### 3.3 向 RAG 记忆的演进预留

1. 将属性定义、会话摘要、关键事实逐步拆分为可检索记忆单元。
2. AI 请求前可按当前联系人注入最相关记忆，而不是一次性塞入全部历史。
3. `distilled_skill` 与后续检索记忆应可以并存，而不是相互替代。

### 3.4 阶段三交付标准

- 联系人画像字段可被编辑并持久保存。
- 重启应用后仍可恢复联系人资料。
- AI 生成链路已接入当前联系人的本地画像数据。

## 阶段四：外部引用与富输入扩展

**目标**：在保持手动主链路稳定的前提下，把输入从纯文本粘贴扩展到多源引用。

### 4.1 链接与引用入口落地

1. 为消息输入框补齐“链接”“引用”两个入口的交互定义。
2. “链接”入口后续优先支持：
   - 企业微信
   - 飞书
   - 其他目标聊天应用
3. “引用”入口后续优先支持：
   - 图片
   - 文件
   - 表情包

### 4.2 富内容输入结构

1. 为文本、图片、文件、引用内容设计统一的数据描述结构。
2. 在 UI 中提供预览、删除、替换等基础操作。
3. AI 请求结构需兼容“文本 + 引用项”的组合输入。

### 4.3 语音与便捷能力预留

1. 底部语音按钮后续可扩展为语音转文本或语音草稿输入。
2. 评估是否增加“粘贴后即时解析”“最近输入历史”“快捷键聚焦输入框”等低摩擦增强能力。
3. 所有新增能力都不能破坏当前手动复制粘贴主链路的稳定性。

### 4.4 阶段四交付标准

- 链接与引用入口具备明确状态与交互定义。
- 输入区可管理多种内容类型。
- 富输入能力接入后，当前文本粘贴路径仍保持顺畅可用。

## Step 2：紧凑版联系人独立工作台 UI 与阶段二主路径

**状态**：**规划中迭代**（见 `Plan/ui.md` 与上文 §2.1–2.3）；历史实现曾包含独立顶栏 Header，需按新规格改造。

**目标**：把界面调整为「紧凑联系人列表（顶部单行 + 新建图标弹窗）+ **无独立顶栏** + 主体 **左预览 / 右候选**（预览顶条展示昵称、属性定义、Skill；候选含 **3～10 条** 与标题/极简）+ 底部 **左右分栏输入** + **预期列下** **设置图标 | 模型 | 语音 | 发送**」。

**实现范围（当前权威）**：

1. 左侧：列表项两行紧凑卡片；**新建**为顶部行内图标，Tooltip 与弹窗字段见 §2.1。
2. 右侧主体：**左**预览列（联系人信息条 + 气泡）、**右**候选列；**取消** `ContactHeader` 式整宽顶栏。
3. 候选区：条数下拉；**「标题 / 极简」**；卡片可点击编辑、复制；不展示「本次交流预期」「已知画像」块。
4. 输入区：**左右分栏**（左对方消息、右预期），默认约三行；发送成功清空 `incomingMessage`；预期标题行 **Close** 清空 `expectation`。
5. 操作区：在**预期列下方**，**设置图标**（打开 API 配置）、模型、语音、发送**右对齐**成组；不在其下展示「请先粘贴…」类整行提示。
6. 联系人画像：**关系/身份并入属性定义**；新建弹窗为昵称 + 属性定义。
7. AI 请求：携带 `suggestionCount`（或等价），Prompt 含「提供 n 条回复消息」。
8. 仍取消单独 `topic` 字段；每联系人独立工作台状态（含条数偏好）。

**验收标准**：

- 左侧顶部仅一行工具区，新建为图标 + 指定 Tooltip 文案；点击弹窗含昵称、属性定义。
- 右侧无双栏之上的整宽顶栏；预览区标题位展示昵称、属性定义（hover 展开更多）、右侧 Skill。
- 候选条数 3～10 可配，发送后请求与 Mock 均按 n 行为目标。
- 候选卡片无预期/画像展示块；点击文案可编辑；可复制当前文案。
- 发送成功生成后清空「联系人发送的内容」；预期保留，且可用手动 Close 清空预期。
- 两输入框 Label 在左、默认三行可见高度。
- 底部控件顺序符合 §2.3。
- `pnpm lint` / `pnpm build`（及涉及 Command 时 `cargo check`）通过。

**历史执行记录**（保留作 diff 参考，不等同于当前规格）：

- 曾实现两行列表项、独立一行 Header、2～3 条固定候选、两段式输入与聊天模拟区；后续需按本节「当前权威」收敛。
