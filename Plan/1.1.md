# 当前最核心的产品需求

后续每个版本都必须先回答：**这个版本最核心要解决的产品问题是什么？**

当前版本的核心产品目标如下：

1. **让用户在真实聊天场景中，低成本拿到可直接发送的 2-3 条候选回复。**
   - 交互仍然坚持“手动触发获取建议”，而不是恢复 Ghost Text 实时自动渲染。
   - UI 的职责是降低操作成本，而不是干扰用户当前聊天节奏。

2. **让 AI 的建议明确围绕“聊天对象”而不是“用户自己”展开。**
   - 产品核心资产是 `contact.skill`。
   - `contact.skill` 由手动标签、历史消息蒸馏结果、上下文摘要共同构成。
   - AI 输出必须持续受聊天对象画像约束。

3. **让桌面端外壳具备稳定可用的基础体验。**
   - 侧边栏窗口可以拖拽、最小化、隐藏、关闭、切换置顶。
   - 即使系统探针或 AI 服务暂时不可用，应用也要保持清晰的降级体验和错误提示。

# 当前技术架构与边界

后续每个版本都必须明确：**当前版本建立在什么已落地架构之上，以及哪些技术边界已经被锁定。**

## 当前已落地的技术基线

- **桌面外壳**：Tauri v2 + Rust
- **前端视图层**：Next.js 16（App Router）+ React 19 + TypeScript
- **样式与组件**：TailwindCSS v4 + Shadcn UI + lucide-react
- **包管理**：pnpm
- **运行模式**：Next.js `output: 'export'` 静态导出，由 Tauri 加载 `out/`
- **窗口形态**：无边框、始终置顶、侧边栏尺寸、自绘拖拽条与窗口控制按钮
- **当前数据来源**：Mock 数据
- **当前 AI 状态**：尚未接入真实模型，仅完成建议卡片 UI 与 Mock 建议生成

## 当前已明确锁定的架构决策

以下技术路线已经明确锁定，并应视为后续版本继续演进时的默认前提：

1. **AI 调用不再采用 Next.js API Routes / Server Actions。**
   - 原因：当前项目已经选择 Next.js 静态导出模式，这条路与运行方式不兼容。

2. **AI 调用统一采用 Tauri Rust Command 中转。**
   - 前端通过 `invoke(...)` 请求 Rust Command。
   - Rust 负责模型请求、密钥管理、代理兼容、错误处理。
   - `contact.skill`、蒸馏、Prompt Builder、Provider Adapter 等中间层能力优先放在 Rust Command 管线中。

3. **网络代理兼容优先在 Rust 网络层解决。**
   - 后续模型请求建议走 Rust `reqwest` / 可控 HTTP 客户端。
   - 目标是兼容本地代理环境（如 Surge）并保留后续插入中间件的能力。

4. **前端职责保持轻量。**
   - 前端主要负责：状态展示、手动触发、候选卡片渲染、用户编辑画像标签、错误反馈。
   - 系统能力、AI 能力、持久化能力均优先经由 Tauri Command 暴露。

## 当前明确不采用的路线

1. **不回退到 Python / PyQt6 / LangChain-Python 技术栈。**
2. **不采用 Next.js Server Actions / API Routes / ISR 作为桌面端 AI 主链路。**
3. **不采用浏览器端直持 API Key 的直连方案作为默认架构。**

# 当前未完成项与未来规划

## 阶段二：AI 中转层与 `contact.skill` 管线落地（当前首要任务）

**目标**：打通“前端触发 -> Rust Command -> 大模型 -> 返回候选建议”的第一条真实 AI 闭环。

### 2.1 Command 化 AI 调用

1. 在 `src-tauri` 中新增建议生成 Command，例如：
   - `generate_suggestions`
2. 前端 `SuggestionPanel` 改为优先调用 Tauri Command，而不是本地 Mock 生成函数。
3. 设计统一的请求 / 响应结构，例如：
   - 请求：`contact`、`intent`、`recent_messages`
   - 响应：`suggestions[]`、调试信息、错误信息

### 2.2 `contact.skill` 结构正式化

1. 将 `contact.skill` 从前端临时类型升级为跨端共享的数据契约。
2. 至少包含以下字段：
   - `manual_tags`
   - `distilled_traits`
   - `last_summary`
   - `updated_at`
3. 明确哪些字段由用户维护，哪些字段由蒸馏器自动生成。

### 2.3 Prompt Builder / Middleware 管线

1. 在 Rust 侧建立建议生成管线：
   - 输入清洗
   - 画像装配
   - 意图装配
   - Prompt Builder
   - Provider Adapter
   - 结果解析与排序
2. 为未来插入以下中间件预留位置：
   - `contact.skill` 蒸馏
   - 历史摘要压缩
   - 风格约束
   - 安全过滤
   - 实验性 rerank / 多模型投票

### 2.4 Provider 与代理兼容

1. 先支持 OpenAI-compatible provider。
2. 第一批兼容目标：
   - DeepSeek
   - 豆包
   - 其他 OpenAI 协议兼容模型
3. API Key 不进入前端渲染层。
4. 明确代理读取策略与超时 / 重试 / 错误映射策略。

### 2.5 阶段二交付标准

完成后，至少应满足：

- 前端点击“获取 AI 建议”可拿到真实模型返回结果。
- 结果条数稳定为 2-3 条。
- 请求失败时前端能展示可理解的错误信息。
- 模型调用链路可插入后续的 `skill` 蒸馏中间件。

## 阶段三：系统探针层（macOS 聊天上下文捕获）

**目标**：从“纯 Mock 驱动”升级到“真实聊天上下文驱动”。

### 3.1 探针能力范围

1. 严格只读，不做任何自动发送或 UI 注入。
2. 在 macOS 环境下尝试获取：
   - 当前聊天对象名称
   - 当前界面的最近几条消息文本
3. 获取失败时必须优雅降级，不得影响应用主界面可用性。

### 3.2 Tauri Command / Event 输出方式

1. 定义探针 Command，例如：
   - `probe_active_chat`
2. 决定前端更新模式：
   - 轮询
   - 事件推送
3. 输出结构应统一，避免前端直接依赖底层系统 API 细节。

### 3.3 阶段三交付标准

- 能在 macOS 上识别目标聊天窗口或明确报告未识别。
- 能获取最近消息或明确报告权限 / 读取失败原因。
- 前端 UI 从 Mock 模式切换到“真实探针数据优先，失败时回退空态/提示态”。

---

## 阶段四：本地画像持久化

**目标**：让 `contact.skill` 成为可积累、可复用的本地长期资产。

### 4.1 本地存储能力

1. 在应用专属目录下读写画像文件。
2. 建议按联系人维度存储，例如：
   - `[contact_name].json`
3. 需要处理：
   - 非法文件名
   - 重名联系人
   - Schema 版本升级

### 4.2 前后端联调闭环

1. 用户编辑手动标签后可落盘。
2. 下次识别到同名联系人时可自动加载画像。
3. AI 建议生成前可读取最新画像。

### 4.3 阶段四交付标准

- 标签修改后可持久保存。
- 重启应用后仍可恢复联系人画像。
- AI 生成链路已接入本地画像读取。

## 阶段五：工程化、可观测性与发布前收口

**目标**：把原型推进到可持续迭代的桌面应用工程形态。

### 5.1 工程质量

1. 为前端与 Rust Command 增加更明确的错误边界。
2. 增加关键模块的类型约束与基础测试。
3. 明确区分开发模式与生产模式日志级别。

### 5.2 用户体验完善

1. 增加“正在捕获 / 未捕获 / 已连接 / AI 请求中 / AI 失败”的状态反馈。
2. 补充加载骨架、空态、失败态。
3. 评估是否增加“复制后反馈”“最近建议历史”“快捷键触发”等轻量增强能力。

### 5.3 打包与分发准备

1. 验证 macOS 打包链路。
2. 校验 Tauri capability、签名、权限提示文案。
3. 为后续内测准备最小可运行发布版本。

# 当前建议的下一步执行顺序

建议后续实现严格按以下顺序推进：

1. **先做阶段二最小闭环**：Rust Command + 单一 provider + 单次返回 2-3 条建议
2. **再做 `contact.skill` 自动蒸馏结构**：把手动标签和自动特征正式区分
3. **再做系统探针**：用真实聊天上下文替换 Mock 数据
4. **最后做本地持久化和工程收口**

这个顺序的原因是：先把“AI 是否真的能生成有价值建议”跑通，再去补系统探针和长期存储，整体风险最低。
