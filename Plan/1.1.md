# 当前最核心的产品需求

后续每个版本都必须先回答：**这个版本最核心要解决的产品问题是什么？**

当前版本的核心产品目标如下：

1. **让用户在无法直接获取聊天上下文的前提下，仍能低成本拿到可直接发送的 2-3 条候选回复。**
   - 当前主路径是：选择联系人 -> 粘贴对方消息 -> 补充本次交流预期 -> 向 AI 请求建议。
   - 交互仍然坚持“手动触发获取建议”，而不是恢复 Ghost Text 实时自动渲染。
   - 当前版本不再把“自动读取聊天窗口”作为近期成立前提，先把手动输入路径做顺滑。

2. **让 AI 的建议明确围绕“聊天联系人”而不是一次性 Prompt 展开。**
   - 左侧联系人列表由用户手动创建和维护。
   - 每个联系人都沉淀名称、属性定义、蒸馏 `skill` 等信息，后续可升级为 RAG 记忆。
   - AI 输出必须持续受联系人画像约束。

3. **让桌面端成为稳定可用的“联系人 + AI 聊天工作台”。**
   - 左侧是联系人列表，右侧是 AI 聊天框。
   - AI 聊天框包含一行联系人 Header、AI 候选回复区、右侧聊天模拟区、两段式输入区与底部操作区。
   - 即使只使用 Mock、或真实模型因配置/网络暂时不可用、或富能力尚未开放，应用也要保持清晰的空态、失败态与可重试体验。

# 当前技术架构与边界

后续每个版本都必须明确：**当前版本建立在什么已落地架构之上，以及哪些技术边界已经被锁定。**

## 当前已落地的技术基线

- **桌面外壳**：Tauri v2 + Rust
- **前端视图层**：Next.js 16（App Router）+ React 19 + TypeScript
- **样式与组件**：TailwindCSS v4 + Shadcn UI + lucide-react
- **包管理**：pnpm
- **运行模式**：Next.js `output: 'export'` 静态导出，由 Tauri 加载 `out/`
- **窗口形态**：无边框、始终置顶、侧边栏尺寸、自绘拖拽条与窗口控制按钮
- **当前数据来源**：联系人等仍为应用内预置/会话态 Mock 数据，尚未做本地文件持久化（见阶段三）
- **当前 AI 状态**：
  - **Mock 路径**：在前端用 `generateMockSuggestions` 生成 2-3 条建议；可在任意环境用于 UI 自测。
  - **真实模型路径**：通过 Tauri Command `generate_ai_suggestions` 调用，Rust 使用 `reqwest` 向 OpenAI-compatible `chat/completions` 发 HTTP 请求；`baseUrl`、`apiKey`、模型名经 `get_ai_config` / `save_ai_config` 读写应用配置目录中的 `ai-provider.json`，不进入前端 bundle。
  - **运行限制**：`pnpm dev` 纯浏览器（非 Tauri）下无法 `invoke` 真实链路，会提示需桌面端；此时应使用 Mock 模型。

## 当前已明确锁定的架构决策

以下技术路线已经明确锁定，并应视为后续版本继续演进时的默认前提：

1. **AI 调用不再采用 Next.js API Routes / Server Actions。**
   - 原因：当前项目已经选择 Next.js 静态导出模式，这条路与运行方式不兼容。

2. **AI 调用统一采用 Tauri Rust Command 中转。**
   - 前端通过 `invoke(...)` 请求 Rust Command。
   - Rust 负责模型请求、密钥管理、代理兼容、错误处理。
   - `contact.skill`、蒸馏、Prompt Builder、Provider Adapter 等中间层能力优先放在 Rust Command 管线中。

3. **网络代理兼容优先在 Rust 网络层解决。**
   - 后续模型请求建议走 Rust `reqwest` / 可控 HTTP 客户端。
   - 目标是兼容本地代理环境（如 Surge）并保留后续插入中间件的能力。

4. **当前主输入链路采用“用户手动复制粘贴”，而不是系统探针 / OCR / 自动读取其他应用。**
   - 前端负责承载结构化输入与粘贴后的上下文整理。
   - 后端 AI 管线围绕联系人信息、对方消息、用户预期组装请求。

5. **联系人是核心组织单位。**
   - 会话记录、属性定义、蒸馏 `skill`、后续 RAG 记忆都优先挂在联系人维度下。
   - 右侧 AI 工作台始终围绕当前选中的联系人展开。

6. **输入能力按“两段式工作流”设计。**
   - 联系人消息粘贴输入框。
   - 用户预期输入框，合并话题背景、对方状态判断、回复目标、语气风格与禁忌点。
   - 不再单独提供 `topic` 字段或话题描述输入框。
   - 为后续链接聊天应用、引用图片 / 文件、语音输入预留扩展位。

## 当前明确不采用的路线

1. **不回退到 Python / PyQt6 / LangChain-Python 技术栈。**
2. **不采用 Next.js Server Actions / API Routes / ISR 作为桌面端 AI 主链路。**
3. **不采用浏览器端直持 API Key 的直连方案作为默认架构。**
4. **当前版本不以系统探针、OCR、剪贴板监听、自动注入聊天应用作为 MVP 主链路。**

# 当前未完成项与未来规划

## 阶段二：联系人工作台与手动上下文输入闭环

**实现状态（与当前仓库一致）**：主界面结构、每联系人独立工作台、Mock 与 Tauri 真实模型两条链路均已落地；**当前版本规划重心已移至阶段三**（本地持久化与画像正式化），本节的条目继续作为**需求规格与回归验收口径**保留。

**目标**：在无法自动读取外站聊天的前提下，把“联系人选择 / 创建 → 粘贴对方消息 → 补充预期 → 生成 2-3 条可发送建议”的闭环定稿并可维护。

### 2.1 左侧联系人列表

1. 支持用户手动创建联系人。
2. 支持从列表中选择当前聊天联系人，并在右侧同步切换工作台。
3. 列表项采用紧凑两行卡片：
   - 第一行：联系人名称 + 最近更新时间
   - 第二行：关系 / 标签，单行截断
   - 不展示多行摘要，确保列表可容纳足够多联系人
4. 为后续扩展预留：
   - 联系人状态 / 标签
   - 最近会话摘要（hover 或展开时展示）

### 2.2 右侧 AI 工作台主结构

1. 顶部信息区：
   - 压缩为一行，展示联系人名称、关系、属性摘要、Skill 摘要。
   - 超出部分省略，鼠标 hover 时展示完整信息（属性定义、notes、tags、traits）。
2. 中间主体区域为左右双栏：
   - 左侧：AI 候选回复区，展示 2-3 条候选回复、加载态、错误态、空态。
   - 右侧：聊天模拟区，把联系人消息和当前选中候选回复以气泡形式预览，帮助用户判断回复是否自然。
3. 每个联系人拥有独立的工作台状态（草稿、候选回复、错误信息），切换联系人时恢复之前的内容。
4. 必须提供空态、加载态、失败态。

### 2.3 两段式输入区与底部操作区

1. 第一输入框：联系人发送的内容。
   - 当前主要承载用户复制粘贴的文本消息。
   - 支持自动撑高，默认高度紧凑。
   - 右下角预留两个入口：
     - 链接
     - 引用
   - 后续可扩展为图片、表情包、文件等内容引用。
2. 第二输入框：我对本次交流的预期。
   - 合并原“本次对话话题描述”与“交流预期”。
   - 用户可自行输入话题背景、对方状态判断、回复目标、语气风格、禁忌点等。
   - 不再单独维护 `topic` 字段。
   - 支持自动撑高，默认高度紧凑。
   - 提供可点击提示词，至少包括：
     - 暧昧
     - 真诚
     - 幽默
     - 严肃
     - 向上管理
     - 普通朋友
     - 委婉拒绝
     - 安抚对方情绪
     - 希望对方合作购买
3. 最底部操作区包含：
   - 左侧：模型选择 / 连接入口
   - 右侧：语音按钮（未来功能）
   - 右侧：发送按钮

### 2.4 与 AI 中转层的对接方式（以当前实现为准）

1. 前端在 **Mock 模式** 下不经过 Rust，直接基于同一套 `AiSuggestionRequest` 生成假数据；在 **OpenAI-compatible 模式** 下调用 Tauri Command `generate_ai_suggestions`。
2. **模型配置**（仅真实链路需要）：`save_ai_config` 写入；`get_ai_config` 读取；未配置时真实链路在命令层返回错误字符串，由右侧工作台展示，输入草稿不丢失。
3. **请求体（`AiSuggestionRequest`，TS/Rust 字段 camelCase 序列化）** 至少包含：
   - `model`：如 `mock`、OpenAI-compatible 时由 UI 与配置共同决定
   - `contact`：当前选中的 `WorkspaceContact` 画像
   - `draft`：`incomingMessage`、用户「交流预期」`expectation`（无独立 `topic` 字段；提示词 chip 以追加文本方式写入 `expectation`）
   - `intent`：由 `expectation` 派生的标签/描述（供 Prompt 与后续演进）
   - `recentMessages`：至少包含对方粘贴内容的一条 `Message` 列表
   - `workspaceContext`：如 `source: "manual-paste"` 与 `generatedAt` 等元数据
4. **成功响应**：`Vec<Suggestion>`，单条为 `{ id, style, text }`。其中 `style` 在解析 OpenAI 响应时由层序占位（如不同候选的区分标签）填充，不依赖多字段 `styleTags`。
5. **失败形态**：Tauri 命令返回 `Err(String)`，前端将字符串写入当前联系人工作台的 `error`；不单独封装 `debug_info` / 顶层 `error` 对象响应体（若后续要可观测性再扩展）。
6. 选择 Mock 时无需 API Key；选择 OpenAI-compatible 时须在桌面端完成一次有效配置，否则发请求会失败并提示，界面保留用户输入。

### 2.5 阶段二交付标准

- 用户可以手动创建联系人并切换当前联系人（创建入口允许为轻量弹窗/表单，以当前实现为准）。
- 切换联系人时，草稿、候选回复、错误信息与「当前预览哪一条候选」独立保留。
- 用户可以完成“粘贴对方消息 +（可选）补充预期 + 发送”的基本流程；**发送的硬性前置条件**为已选联系人与非空的「联系人发送的内容」；交流预期为强烈建议、非硬性必填（与 `canSend` 逻辑一致）。
- 右侧工作台完整体现最新 UI 结构（一行联系人 Header、AI 候选回复区、右侧聊天模拟区、两段式输入区）。
- 聊天模拟区可预览联系人消息与当前选中候选回复；候选卡片可切换预览。
- 在 Mock 或已配置完成的 OpenAI-compatible 下，发送后可获得 2-3 条候选回复；失败时得到可读错误并允许重试，当前输入不丢失。

## 阶段三：`contact.skill` 正式化与本地持久化

**目标**：让联系人画像从展示字段变成可积累、可复用的本地长期资产。

### 3.1 联系人画像结构正式化

1. 将联系人数据从前端临时类型升级为跨端共享的数据契约。
2. 至少包含以下字段：
   - `name`
   - `attribute_definition`
   - `manual_tags`
   - `distilled_skill`
   - `notes`
   - `updated_at`
3. 明确哪些字段由用户维护，哪些字段由蒸馏器自动生成。

### 3.2 本地存储能力

1. 在应用专属目录下读写联系人数据。
2. 建议按联系人维度存储，例如：
   - `[contact_name].json`
3. 需要处理：
   - 非法文件名
   - 重名联系人
   - Schema 版本升级

### 3.3 向 RAG 记忆的演进预留

1. 将属性定义、会话摘要、关键事实逐步拆分为可检索记忆单元。
2. AI 请求前可按当前联系人注入最相关记忆，而不是一次性塞入全部历史。
3. `distilled_skill` 与后续检索记忆应可以并存，而不是相互替代。

### 3.4 阶段三交付标准

- 联系人画像字段可被编辑并持久保存。
- 重启应用后仍可恢复联系人资料。
- AI 生成链路已接入当前联系人的本地画像数据。

## 阶段四：外部引用与富输入扩展

**目标**：在保持手动主链路稳定的前提下，把输入从纯文本粘贴扩展到多源引用。

### 4.1 链接与引用入口落地

1. 为消息输入框补齐“链接”“引用”两个入口的交互定义。
2. “链接”入口后续优先支持：
   - 企业微信
   - 飞书
   - 其他目标聊天应用
3. “引用”入口后续优先支持：
   - 图片
   - 文件
   - 表情包

### 4.2 富内容输入结构

1. 为文本、图片、文件、引用内容设计统一的数据描述结构。
2. 在 UI 中提供预览、删除、替换等基础操作。
3. AI 请求结构需兼容“文本 + 引用项”的组合输入。

### 4.3 语音与便捷能力预留

1. 底部语音按钮后续可扩展为语音转文本或语音草稿输入。
2. 评估是否增加“粘贴后即时解析”“最近输入历史”“快捷键聚焦输入框”等低摩擦增强能力。
3. 所有新增能力都不能破坏当前手动复制粘贴主链路的稳定性。

### 4.4 阶段四交付标准

- 链接与引用入口具备明确状态与交互定义。
- 输入区可管理多种内容类型。
- 富输入能力接入后，当前文本粘贴路径仍保持顺畅可用。

# 当前建议的下一步执行顺序

与当前代码对齐后的优先级如下：

1. **阶段三：联系人画像与本地持久化**（当前首要增量）——将联系人、工作台草稿等从仅内存/预置数据升级为应用目录下的可恢复资产，并明确跨版本 Schema 策略。
2. **在阶段三基础上打磨阶段五部分条目**：更清晰的错误码/类型边界、基础测试、与真实发送路径相关的状态反馈，便于持续迭代与发布前收口。
3. **阶段四：链接 / 引用 / 富输入与语音**——在手动主路径稳定、数据可持久化的前提下，再扩展输入源与多模态结构。
4. **持续维护阶段二产物的回归**：UI 与 `generate_ai_suggestions` 契约若变更，需同步更新本文件与 `Plan/ui.md` 中的规格描述。

原「先做界面、再接 Rust」的顺序已在当前仓库中完成，无需重复执行；后续风险主要来自**数据不落地**时无法形成长期用户资产，故把持久化提前为下一主里程碑。
