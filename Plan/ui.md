# Social Copilot UI 草图

> 本文档基于 `Plan/1.1.md` 生成，并与 `src/lib/types.ts` 及当前主界面实现保持同一步调。
> 当前 UI 目标是：在无法直接获取用户聊天信息的前提下，通过“联系人列表 + 手动复制粘贴消息 + AI 回复建议 + 聊天模拟预览”的方式，让用户低成本获得可直接发送的回复。

---

# 1. UI 总体结构

当前主界面采用左右分栏：

1. **左侧：聊天联系人列表**
   - 用户自定义创建联系人。
   - 用户从列表中选择一个联系人。
   - 当前选中的联系人决定右侧 AI 工作台上下文。
   - 每个联系人拥有独立的输入草稿、候选回复与错误状态。

2. **右侧：AI 聊天工作台**
   - **无**独立整宽顶栏；联系人昵称、属性定义、Skill 出现在**聊天气泡列（主体双栏之左列）顶部**。
   - 中间主体为左右双栏：**左侧**为聊天预览区（顶条 + 对方/本人气泡）；**右侧**为 AI 候选回复区（含 **3～10 条** 下拉与 **标题/极简** 分段控件，**紧凑排版**）。
   - 底部为**左右两列**：左列「联系人发送的内容」，右列「我对本次交流的预期」（chip 在右列）；**设置图标 + 模型下拉 + 语音 + 发送**置于**右列输入区下方**、右对齐（设置打开原 Provider 配置弹窗）。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Social Copilot                                                              │
├───────────────────────┬──────────────────────────────────────────────────────┤
│ 联系人        搜索 [+] │ （无双栏顶上的整行 Header）                            │
│ （顶部一行；+ 为新建）  ├──────────────────────────────┬───────────────────────┤
│                       │ 张三  属性定义…  Skill…       │ [回复条数▼][标题|极简] │
│ 张三        昨天       │ 对方 / 我 气泡（预览在左）     │ AI 候选列表（配置在右） │
│ 属性定义摘要…         │                              │                       │
│                       ├──────────────────────────────┴───────────────────────┤
│ 李经理      10:31      │ 底：左「对方消息」| 右「预期」+ chip；右下 [⚙][模型▼][语音][发送] │
│ 属性定义摘要…         │                                                      │
└───────────────────────┴──────────────────────────────────────────────────────┘
```

# 2. 左侧联系人列表

## 2.1 信息架构

左侧列表负责组织用户的聊天对象。列表项必须紧凑，优先展示更多联系人。**列表顶部工具区只占一行**，不与下方联系人卡片混为「两行一条」。

```text
┌───────────────────────┐
│ 联系人      搜索  [+] │  ← 一行：[+] 为图标，hover 见下
├───────────────────────┤
│ 张三             昨天  │
│ 属性定义摘要（单行）   │
│                       │
│ 李经理           10:31 │
│ 属性定义摘要         │
│                       │
│ 王客户           周一  │
│ 属性定义摘要         │
└───────────────────────┘
```

说明：原「普通朋友 / 向上管理」等**身份或关系语义**一律写入**属性定义**；列表第二行展示的是属性定义摘要，不再单独强调「关系」字段。

## 2.2 核心组件

- **新建联系人（图标）**
  - 位于顶部行**右侧**（或约定位置），**仅占图标区域**，不占用下方列表项的两行高度。
  - **Tooltip 标题**：`新建联系人`
  - **Tooltip 描述**：`每个新联系人会独享一个聊天对话上下文`
  - 点击打开**弹窗**：输入**昵称**、**属性定义**（多行/长文本由实现决定；创建后可在阶段三编辑持久化）。

- **搜索联系人输入框**
  - 与标题、新建图标**同一顶行**内布局（具体栅格以实现为准）。
  - 按联系人名称搜索；后续可扩展按属性定义关键词搜索。

- **联系人列表项**
  - 固定两行展示（**仅列表项**，不含顶栏）。
  - 第一行：联系人名称 + 最近更新时间。
  - 第二行：**属性定义**摘要，单行截断。
  - 当前选中项需要高亮。

## 2.3 状态

- **空态**
  - 无联系人时展示“创建第一个联系人”。

- **选中态**
  - 当前联系人高亮。
  - 右侧工作台同步切换。
  - 切换联系人时恢复该联系人的独立草稿、候选回复与错误状态。

- **搜索无结果态**
  - 展示“没有找到联系人”。
  - 保留“新建联系人”入口。

# 3. 右侧 AI 聊天工作台

右侧 AI 工作台由 **3** 个主区域组成（**无**跨双栏的独立顶栏）：

1. 主体双栏：**左**聊天预览区（顶条嵌联系人信息）+ **右** AI 候选回复区（条数、标题/极简 + 建议列表）
2. 底部**左右分栏**输入区：左「联系人发送的内容」、右「我对本次交流的预期」（约三行高；右列含 chip）
3. 操作条在**右列下方**：**设置图标**（打开 API/Provider 配置弹窗）**紧挨**模型下拉，其次语音、发送，组内**右对齐**

```text
┌────────────────────────────────────────────────────────────────────┐
│ 张三 属性… Skill │ [条数 3▼…10]  [标题|极简]                           │
├──────────────────┼────────────────────────────────────────────────────┤
│ 对方 / 我 气泡    │ 建议 1… 建议 2…                                    │
├──────────────────┴────────────────────────────────────────────────────┤
│ 联系人发送的内容（左栏）    │ 我对本次交流的预期 [✕]（右栏）            │
│ [≈三行…]                   │ [≈三行…]  [chip…]                        │
│                             │ [⚙] [模型▼] [语音] [发送]                │
└────────────────────────────────────────────────────────────────────┘
```

# 4. 预览区顶条（取代原「聊天模拟」标题行）

## 4.1 布局与语义

原右侧「聊天模拟」**标题那一行**改为**当前联系人上下文条**（**单行紧凑**：较小字号；**整行垂直居中**），从左到右建议为：

1. **昵称**（左，固定列）
2. **属性定义**（中，摘要截断；**hover** 时 `title` 展示**完整属性定义**及可用的 `summary` 等）
3. **Skill**（右，截断；**hover** `title` 展示标签、蒸馏特征、notes 全文）。无有效 skill 数据且非错误态时文案为 **「待上传蒸馏 skill」**；`skillFileError === true` 时为 **「skill 信息错误」**

下方仍为聊天气泡预览（对方消息 + 当前选中建议）。

## 4.2 Hover

- 鼠标移到**中间属性定义列**时，通过原生 `title` 展示**完整属性定义**及 `summary`（若有）；不将 notes/tags/traits 混进属性 `title`（Skill 列单独展示画像全文）。
- **Skill 列** hover 展示该联系人 `skill` 的摘要全文；空数据与错误态的说明见 §4.1。

## 4.3 与旧版差异

- **删除**跨整个右侧面板最上方的「一行 Header」；信息收敛到**预览列顶条**，与模拟对话同一列，减少垂直占用。

# 5. 主体双栏区域

## 5.1 AI 候选回复区

**主体双栏中的右列**展示 AI 辅助生成结果（左列为聊天预览，见 §5.3）。

**顶部控件**：

- **回复条数**下拉：**3～10**，默认 **3**；与发送请求绑定，并在 Prompt 中体现「请提供 **n** 条回复消息」类约束（见 `Plan/1.1.md` §2.4）。
- **「标题 / 极简」Switch**（或分段按钮）：与条数下拉同一工具行，**控件与字号偏小**，减少占用高度。
- **「标题 / 极简」语义**：**标题**：显示 `style`（若有）作为卡片**紧凑**标题区，下为可编辑正文（较常规 UI **再小两个字号**，如 **10px** 档）；**极简**：**完全不展示** `style`，**复制按钮**置于**正文行最左侧**。正文 `textarea` **高度随内容**，**单行时不保留一整行空白底**。**默认选中「标题」**；按**联系人**记忆（写入 `ContactWorkspaceState`）。
- 列表滚动容器保留**内边距**，选中态建议 **ring-inset**，避免首条卡片**上边框**与**左侧边框**被父级裁切。

主要展示：

- AI 生成的 **n** 条候选回复
- 加载态、错误态、空态
- 后续多轮重新生成记录

不再常驻展示“本次上下文”摘要卡片；**不在卡片内展示「本次交流预期」「已知画像」等块**，避免干扰复制发送。**Mock** 路径返回的 `text` 亦**不**夹带 `[Mock · 意图=…]`、`已知画像` 等调试可见串（与 `Plan/1.1.md` 技术基线一致）。

## 5.2 候选回复卡片

每条候选回复在数据上对应 `Suggestion`：`id`、`style`（字符串）、`text`。

**与「标题 / 极简」Switch 联动**：

- **标题**：卡片上部紧凑展示 `style`（无则仅保留复制与正文区），下方为 **10px** 档可编辑正文，`textarea` **自适应行高**；卡片整体**低内边距**。
- **极简**：**不展示**任何 `style` 文案；首列为**复制**，其后为正文；版面以 `text` 为唯一视觉主体；字号与标题模式正文一致。

通用交互：

- **点击正文区域**：进入可编辑状态（或内联输入），用户可基于 AI 改写或完全自拟；**复制**复制**当前展示的最终字符串**（含用户修改后）。
- 复制按钮（或快捷键）与点击编辑行为可并存。
- 微调 / 反馈按钮可暂缓。

建议风格：

- 卡片轻量；重点「一键复制 / 点改再复制」。
- 避免聊天机器人式多段元数据（预期、画像等仍不出现）。

## 5.3 聊天预览（模拟）区

**主体双栏中的左列**用于模拟真实聊天窗口中的对话效果；**顶条**为 §4 所述昵称 + 属性定义 + Skill，**不再**单独写死「聊天模拟」四字标题（若需无障碍 `aria-label` 可保留隐藏文案）。

目标：

- 判断候选回复放入上下文后是否自然；语气、长度是否合适。

第一版展示：

- 顶条：昵称、属性定义（摘要）、Skill（见 §4.1 空/错态文案）。
- 对方消息气泡：来自「联系人发送的内容」。
- 右侧「我」气泡：当前选中候选的**当前文案**（若用户在卡片中编辑过，与卡片一致）；气泡正文与候选卡片同为 **10px** 档、**紧凑内边距**。
- **对方空态**：仅当**尚无候选回复**且**对方输入框为空**时，展示「粘贴联系人消息后…」引导；**一旦左侧已有候选列表**（例如发送成功后已清空对方输入），**不再展示**该引导，避免与「已生成建议」状态冲突。
- 无候选时：空态或「生成后将在这里预览」。

```text
┌─────────────────────────────┐
│ 张三  属性定义摘要…    Skill │
├─────────────────────────────┤
│ 对方                         │
│ 这周有点累，可能不太想出门。 │
│                         我  │
│ 没关系，你先好好休息...      │
└─────────────────────────────┘
```

## 5.4 状态

- **未选择联系人**
  - 提示用户从左侧选择或创建联系人。

- **未输入消息**
  - AI 候选回复区提示用户粘贴联系人发来的内容。
  - 聊天模拟区：若**也无候选回复**，展示对方粘贴引导；若**已有候选**，不展示该引导（见 §5.3）。

- **AI 请求中**
  - AI 候选回复区展示加载骨架或生成中状态。
  - 聊天模拟区可继续展示当前输入。

- **生成失败**
  - 展示错误信息。
  - 保留当前联系人下的输入内容。
  - 允许重试。

- **已生成建议**
  - AI 候选回复区展示 **n** 条候选回复（与下拉一致）。
  - 预览区展示当前预览回复。

# 6. 底部两段式输入区

底部输入区是当前版本最重要的交互区域。当前正式取消单独的 `topic` 字段。

**版式**：**左右两列栅格**——左列仅「联系人发送的内容」（标题在上，多行输入，**默认约三行**）；右列「我对本次交流的预期」（标题行含 **✕** 清空，多行输入，**chip 在输入框下**）。**设置图标 + 模型下拉 + 语音 + 发送**放在**右列底部**、右对齐（设置即原「配置模型」弹窗入口）。

```text
┌────────────────────────────┬─────────────────────────────────────────┐
│ 联系人发送的内容            │ 我对本次交流的预期 [✕]                   │
│ ┌────────────────────────┐ │ ┌─────────────────────────────────────┐   │
│ │ ≈三行；粘贴对方消息…    │ │ │ ≈三行；预期、语气、禁忌…            │   │
│ │           [链接][引用]│ │ └─────────────────────────────────────┘   │
│ └────────────────────────┘ │ [chip…]                                  │
│                            │ [⚙] [模型▼] [语音] [发送]                 │
└────────────────────────────┴─────────────────────────────────────────┘
```

# 7. 输入框 1：联系人发送的内容

## 7.1 目的

承载用户复制粘贴来的对方消息。

当前版本主要支持：

- 文本
- 多段文本
- 简单换行

未来支持：

- 图片
- 表情包
- 文件
- 来自企业微信 / 飞书等应用的链接引用

## 7.2 交互

- 必填。
- 工作台底部为**左右分栏**时，该字段在**左列**，标题在输入区**上方**（与实现一致）。
- **默认约三行**可见高度；支持自动撑高至最大高度。
- 输入框右下角展示两个 icon：
  - 链接
  - 引用

推荐 placeholder：

```text
粘贴对方发来的消息。
```

（实现中**避免**「可以是一句话，也可以是多段聊天记录」等赘述。）

## 7.3 链接 icon

当前状态：

- 可以展示为禁用或“即将支持”。

未来含义：

- 链接企业微信、飞书等聊天应用。
- 通过用户授权或手动选择，把某个外部会话作为上下文来源。

## 7.4 引用 icon

当前状态：

- 可以展示为禁用或“即将支持”。

未来含义：

- 引用图片、文件、表情包等内容。
- 引用内容以附件卡片形式出现在输入框内部或下方。

# 8. 输入框 2：我对本次交流的预期

## 8.1 目的

让用户明确告诉 AI：这次回复想达到什么效果。该输入框同时承载原本的“话题描述”职责。

用户可以输入：

- 本次话题背景
- 对方可能的想法或状态
- 我希望回复达到的效果
- 回复语气 / 风格
- 不希望出现的表达
- 需要避免踩的雷

示例：

```text
这是项目延期沟通，我希望显得负责但不要背锅。对方可能有点不满，语气要真诚、稳一点，不要太防御。
```

## 8.2 交互

- 建议强提示填写，但不作为硬性必填。
- 位于底部**右列**；标题在输入区**上方**，标题行右侧为 **Close**；**默认约三行**可见高度。
- **右侧**（推荐与 Label 同一行右对齐，或紧贴输入区右上角）放置 **Close 图标**：点击后清空 `expectation`，需有 `aria-label`（如「清空交流预期」）；可选二次确认由产品决定，默认**无需**确认以保流畅。
- 下方提供提示词 chip。
- 点击 chip 后，把对应词语插入输入框或追加为目标标签。

推荐 placeholder：

```text
告诉 AI 这次你想怎么回复：可以写话题背景、你猜测对方的想法、希望达到的效果、语气风格、不要踩的雷。
例如：这是项目延期沟通，我希望显得负责但不要背锅；对方可能有点不满，语气要真诚、稳一点。
```

## 8.3 提示词 chip

第一批提示词（与 `page.tsx` 中 `PROMPT_CHIPS` 一致）：

- 暧昧
- 真诚
- 幽默
- 严肃
- 向上管理
- 普通朋友
- 委婉拒绝
- 安抚对方情绪
- 希望对方合作购买

点击行为：若当前预期中尚未包含该词，则将其**追加**到「我对本次交流的预期」文本（可选前加空格）；**不**单独维护 `selectedPromptChips` 数组（与当前 `ComposerDraft` 一致）。

# 9. 底部操作区

## 9.1 布局

操作区固定在**「我对本次交流的预期」列底部**（与左列「联系人发送的内容」并排时，整体仍靠工作台右下）。组内**从左到右**顺序为：

1. **设置图标**（打开 Provider / API 配置弹窗，等同原「配置模型」）
2. **模型下拉**（Mock / OpenAI-compatible 等），与设置图标**同一行紧邻**
3. **语音**（未来）
4. **发送**

窄屏下输入区可改为上下堆叠，操作条仍跟在预期列下方。

## 9.2 模型选择

当前支持：

- **Mock 模型**：纯前端 `generateMockSuggestions`，无需配置文件；适合开发与浏览器静态预览（非 Tauri 时仅能走此路）。
- **OpenAI-compatible**：在 **Tauri 桌面端** 经 `generate_ai_suggestions` 发 HTTP 请求；需通过「配置模型」在应用配置目录保存 Base URL、API Key 与模型名（与 `Plan/1.1.md` 技术基线一致）。

未在桌面端成功保存有效配置时，若仍选择 OpenAI-compatible 并点击发送，请求会在收到错误响应后于工作台显示失败原因，输入内容保留，引导用户打开「配置模型」或临时切回 Mock。

## 9.3 语音按钮

当前定位为未来功能。

当前 UI 表现：

- 可以展示但禁用。
- hover 或点击提示“语音输入即将支持”。

未来能力：

- 语音转文字
- 口述本次预期
- 口述回复风格

## 9.4 发送按钮

点击后将当前结构化输入发送给 AI 模型。

发送前校验（与 `sendDisabledReason` 一致）：

- 已选择联系人
- 已输入「联系人发送的内容」非空

说明：是否「已连上真实模型」**不作为**禁止点击发送的前置条件；Mock 直接成功；OpenAI-compatible 在配置缺失或网络/HTTP 错误时，在请求结束后展示错误。交流预期为强烈建议填写，不阻塞发送。

**不在**底栏下方单独展示「请先粘贴联系人发送的内容」等整行说明；未满足条件时依赖**发送按钮禁用**及其 **`title` / `aria` 提示**即可。

发送后行为：

- AI 候选回复区展示生成中状态。
- AI 返回后展示 **n** 条候选回复。
- 预览区展示当前候选回复。
- **成功完成一次生成请求后**：清空「联系人发送的内容」；**保留**「我对本次交流的预期」。预期行**右侧**（与 Label 同行或输入区右上方，以实现为准）提供 **Close（✕）图标**，一键清空预期全文，与是否刚发送无关。
- 失败时保留两段输入并允许重试。

## 12.2 WorkspaceContact

当前主界面使用的联系人。**产品语义**：凡原「关系 / 身份」（如普通朋友）均并入 **`attributeDefinition`**；`relation` / `summary` 若仍存在于类型中，视为兼容字段或列表摘要的派生，**UI 与新建弹窗以「昵称 + 属性定义」为准**。

```ts
type WorkspaceContact = Contact & {
  /** @deprecated 逐步由 attributeDefinition 吸收；列表第二行展示属性定义摘要 */
  relation?: string
  attributeDefinition: string
  lastActive: string
  /** 可与 attributeDefinition 摘要合并展示 */
  summary?: string
```

已正式取消单独的 `topic` 字段。提示词 chip 只通过修改 `expectation` 字符串体现，不另存数组或附件。

```ts
type ContactWorkspaceState = {
```

# 13. 状态矩阵

| 场景 | 左侧联系人列表 | 右侧工作台 | 发送按钮 |
| --- | --- | --- | --- |

# 14. 视觉方向

## 14.1 整体风格

- 桌面工具感
- 干净、轻量、低干扰
- 左侧偏导航，右侧偏工作区
- 候选回复突出可复制性
- 聊天模拟区突出“放进真实聊天里的效果”

## 14.2 布局建议

- 左侧宽度：240-280px
- 右侧自适应
- 预览列顶条高度：与一行工具条相当（约 36-44px）
- 中间主体双栏自适应撑满
- 预览列宽度：280-360px
- 底部输入区根据输入内容自动增高，但需要设置最大高度
- 两个输入框**默认约三行**基准高度，再随内容增长

## 14.3 颜色建议

- 背景：浅灰或深色模式均可支持
- 当前联系人：使用轻微高亮背景
- 发送按钮：主品牌色
- 禁用未来功能：低对比度 + tooltip
- AI 建议卡片：白底 / 深色卡片 + 轻阴影
- 聊天模拟区：接近真实聊天气泡，但保持轻量工具感
