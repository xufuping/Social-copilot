# Social Copilot UI 草图

> 本文档基于 `Plan/1.1.md` 生成，并与 `src/lib/types.ts` 及当前主界面实现保持同一步调。
> 当前 UI 目标是：在无法直接获取用户聊天信息的前提下，通过“联系人列表 + 手动复制粘贴消息 + AI 回复建议 + 聊天模拟预览”的方式，让用户低成本获得可直接发送的回复。

---

# 1. UI 总体结构

当前主界面采用左右分栏：

1. **左侧：聊天联系人列表**
   - 用户自定义创建联系人。
   - 用户从列表中选择一个联系人。
   - 当前选中的联系人决定右侧 AI 工作台上下文。
   - 每个联系人拥有独立的输入草稿、候选回复与错误状态。

2. **右侧：AI 聊天工作台**
   - 顶部使用一行联系人信息 Header。
   - 中间主体采用左右双栏：左侧 AI 候选回复区，右侧聊天模拟区。
   - 底部提供两段式输入框与操作按钮。

```text
┌──────────────────────────────────────────────────────────────────────────────┐
│ Social Copilot                                                              │
├───────────────────────┬──────────────────────────────────────────────────────┤
│ 左侧联系人列表         │ 一行联系人 Header                                    │
│                       ├──────────────────────────────┬───────────────────────┤
│ + 新建联系人           │ AI 候选回复区                  │ 聊天模拟区             │
│ 搜索联系人             │                              │                       │
│                       │ 候选回复 / 加载 / 错误 / 空态   │ 对方消息 / 我的回复预览 │
│ 张三        昨天       │                              │                       │
│ 普通朋友 · 真诚        ├──────────────────────────────┴───────────────────────┤
│                       │ 底部两段式输入区 + 操作区                              │
│ 李经理      10:31      │                                                      │
│ 向上管理 · 结构化      │                                                      │
└───────────────────────┴──────────────────────────────────────────────────────┘
```

# 2. 左侧联系人列表

## 2.1 信息架构

左侧列表负责组织用户的聊天对象。列表项必须紧凑，优先展示更多联系人。

```text
┌───────────────────────┐
│ 联系人            [+]  │
│ 搜索联系人             │
│                       │
│ 张三             昨天  │
│ 普通朋友 · 慢热 · 真诚 │
│                       │
│ 李经理           10:31 │
│ 向上管理 · 结构化      │
│                       │
│ 王客户           周一  │
│ 客户 · ROI · 信任      │
└───────────────────────┘
```

## 2.2 核心组件

- **新建联系人按钮**
  - 点击后创建联系人。
  - 可先使用弹窗或右侧空态表单。

- **搜索联系人输入框**
  - 按联系人名称搜索。
  - 后续可扩展按标签、关系、最近会话搜索。

- **联系人列表项**
  - 固定两行展示。
  - 第一行：联系人名称 + 最近更新时间。
  - 第二行：关系 / 标签 / 简短状态，单行截断。
  - 不展示多行摘要，摘要可后续放到 hover 或详情中。
  - 当前选中项需要高亮。

## 2.3 状态

- **空态**
  - 无联系人时展示“创建第一个联系人”。

- **选中态**
  - 当前联系人高亮。
  - 右侧工作台同步切换。
  - 切换联系人时恢复该联系人的独立草稿、候选回复与错误状态。

- **搜索无结果态**
  - 展示“没有找到联系人”。
  - 保留“新建联系人”入口。

# 4. 一行联系人 Header

## 4.1 布局

顶部信息区只占用一行文字高度，展示当前联系人最关键的信息。

```text
张三 · 普通朋友    慢热 / 喜欢轻松表达 / 最近工作压力偏大    Skill: 真诚、安抚、给选择空间
```

## 4.2 默认展示内容

- 联系人名称
- 关系类型
- 属性定义摘要
- Skill 摘要

展示规则：

- 单行展示。
- 超出宽度时截断。
- 不使用大卡片。
- 不在默认态展示 notes、完整属性、完整 traits。

## 4.3 Hover 展示完整信息

鼠标移动到 Header 上时展示更多信息：

- 联系人属性定义全文
- 用户 notes
- manual tags
- distilled traits
- 最近更新时间

当前版本可先使用浏览器原生 `title` 或轻量 tooltip。后续可替换为 Popover。

# 5. 主体双栏区域

## 5.1 AI 候选回复区

左侧区域展示 AI 辅助生成结果。

主要展示：

- AI 生成的候选回复
- 加载态
- 错误态
- 空态
- 后续多轮重新生成记录

不再常驻展示“本次上下文”摘要卡片，避免重复占用空间。

## 5.2 候选回复卡片

每条候选回复在数据上对应 `Suggestion`：单条**风格/标签**字段 `style`（字符串）、正文 `text`、稳定键 `id`。

界面上至少包含：

- 回复正文
- 与 `style` 对应的展示（作卡片标题区）
- 复制按钮
- 微调按钮（可暂缓）
- 反馈按钮（可暂缓）

建议风格：

- 卡片轻量、留白适中。
- 重点突出“可直接复制发送”。
- 避免把界面做成复杂聊天机器人。

## 5.3 聊天模拟区

右侧区域用于模拟真实聊天窗口中的对话效果。

目标：

- 让用户判断候选回复放入上下文后是否自然。
- 让用户直观看到语气是否太硬、太软、太长或不协调。
- 降低复制到真实聊天软件前的不确定性。

第一版展示：

- 联系人发送的内容：左侧气泡。
- 当前选中候选回复：右侧气泡。
- 没有候选回复时，展示“生成后将在这里预览”。
- 默认可预览第一条候选回复，后续支持点击候选卡片切换预览。

```text
┌─────────────────────────────┐
│ 聊天模拟                     │
│                             │
│ 对方                         │
│ 这周有点累，可能不太想出门。 │
│                             │
│                         我  │
│ 没关系，你先好好休息...      │
└─────────────────────────────┘
```

## 5.4 状态

- **未选择联系人**
  - 提示用户从左侧选择或创建联系人。

- **未输入消息**
  - AI 候选回复区提示用户粘贴联系人发来的内容。
  - 聊天模拟区展示空态。

- **AI 请求中**
  - AI 候选回复区展示加载骨架或生成中状态。
  - 聊天模拟区可继续展示当前输入。

- **生成失败**
  - 展示错误信息。
  - 保留当前联系人下的输入内容。
  - 允许重试。

- **已生成建议**
  - AI 候选回复区展示 2-3 条候选回复。
  - 聊天模拟区展示当前预览回复。

# 7. 输入框 1：联系人发送的内容

## 7.1 目的

承载用户复制粘贴来的对方消息。

当前版本主要支持：

- 文本
- 多段文本
- 简单换行

未来支持：

- 图片
- 表情包
- 文件
- 来自企业微信 / 飞书等应用的链接引用

## 7.2 交互

- 必填。
- 支持自动撑高。
- 默认高度紧凑，避免占用过多垂直空间。
- 输入框右下角展示两个 icon：
  - 链接
  - 引用

推荐 placeholder：

```text
粘贴对方刚刚发来的消息。可以是一句话，也可以是多段聊天记录。
```

## 7.3 链接 icon

当前状态：

- 可以展示为禁用或“即将支持”。

未来含义：

- 链接企业微信、飞书等聊天应用。
- 通过用户授权或手动选择，把某个外部会话作为上下文来源。

## 7.4 引用 icon

当前状态：

- 可以展示为禁用或“即将支持”。

未来含义：

- 引用图片、文件、表情包等内容。
- 引用内容以附件卡片形式出现在输入框内部或下方。

---

# 8. 输入框 2：我对本次交流的预期

## 8.1 目的

让用户明确告诉 AI：这次回复想达到什么效果。该输入框同时承载原本的“话题描述”职责。

用户可以输入：

- 本次话题背景
- 对方可能的想法或状态
- 我希望回复达到的效果
- 回复语气 / 风格
- 不希望出现的表达
- 需要避免踩的雷

示例：

```text
这是项目延期沟通，我希望显得负责但不要背锅。对方可能有点不满，语气要真诚、稳一点，不要太防御。
```

## 8.2 交互

- 建议强提示填写，但不作为硬性必填。
- 支持自动撑高。
- 默认高度紧凑。
- 下方提供提示词 chip。
- 点击 chip 后，把对应词语插入输入框或追加为目标标签。

推荐 placeholder：

```text
告诉 AI 这次你想怎么回复：可以写话题背景、你猜测对方的想法、希望达到的效果、语气风格、不要踩的雷。
例如：这是项目延期沟通，我希望显得负责但不要背锅；对方可能有点不满，语气要真诚、稳一点。
```

## 8.3 提示词 chip

第一批提示词（与 `page.tsx` 中 `PROMPT_CHIPS` 一致）：

- 暧昧
- 真诚
- 幽默
- 严肃
- 向上管理
- 普通朋友
- 委婉拒绝
- 安抚对方情绪
- 希望对方合作购买

点击行为：若当前预期中尚未包含该词，则将其**追加**到「我对本次交流的预期」文本（可选前加空格）；**不**单独维护 `selectedPromptChips` 数组（与当前 `ComposerDraft` 一致）。

# 9. 底部操作区

## 9.1 布局

底部操作区固定在两段式输入区最下方。

左侧是模型选择与配置入口，右侧是语音与发送。

```text
┌────────────────────────────────────────────────────────────────────┐
│ 模型：Mock / OpenAI-compatible             [配置模型] [语音] [发送] │
└────────────────────────────────────────────────────────────────────┘
```

## 9.2 模型选择

当前支持：

- **Mock 模型**：纯前端 `generateMockSuggestions`，无需配置文件；适合开发与浏览器静态预览（非 Tauri 时仅能走此路）。
- **OpenAI-compatible**：在 **Tauri 桌面端** 经 `generate_ai_suggestions` 发 HTTP 请求；需通过「配置模型」在应用配置目录保存 Base URL、API Key 与模型名（与 `Plan/1.1.md` 技术基线一致）。

未在桌面端成功保存有效配置时，若仍选择 OpenAI-compatible 并点击发送，请求会在收到错误响应后于工作台显示失败原因，输入内容保留，引导用户打开「配置模型」或临时切回 Mock。

## 9.3 语音按钮

当前定位为未来功能。

当前 UI 表现：

- 可以展示但禁用。
- hover 或点击提示“语音输入即将支持”。

未来能力：

- 语音转文字
- 口述本次预期
- 口述回复风格

## 9.4 发送按钮

点击后将当前结构化输入发送给 AI 模型。

发送前校验（与 `sendDisabledReason` 一致）：

- 已选择联系人
- 已输入「联系人发送的内容」非空

说明：是否「已连上真实模型」**不作为**禁止点击发送的前置条件；Mock 直接成功；OpenAI-compatible 在配置缺失或网络/HTTP 错误时，在请求结束后展示错误。交流预期为强烈建议填写，不阻塞发送。

发送后行为：

- 中间 AI 候选回复区展示生成中状态。
- AI 返回后展示 2-3 条候选回复。
- 聊天模拟区预览当前候选回复。
- 失败时保留当前联系人下的输入内容并允许重试。

# 10. 关键用户流程

## 10.1 首次使用流程

```text
打开应用
  ↓
左侧为空态：创建第一个联系人
  ↓
输入联系人名称（当前实现为轻量命名入口；完整画像见阶段三）
  ↓
进入该联系人的 AI 工作台
  ↓
粘贴对方消息
  ↓
（建议）填写交流预期或使用提示词 chip
  ↓
选择模型：Mock 可零配置；OpenAI-compatible 在桌面端打开「配置模型」并保存
  ↓
发送给 AI
  ↓
获得候选回复
  ↓
在聊天模拟区检查效果
  ↓
复制满意回复到真实聊天应用
```

## 10.2 日常使用流程

```text
打开应用
  ↓
从左侧选择联系人
  ↓
恢复该联系人的独立工作台状态
  ↓
粘贴对方最新消息
  ↓
填写本次交流预期
  ↓
发送
  ↓
查看候选回复与聊天模拟
  ↓
复制候选回复
```

## 10.3 快速闲聊流程

```text
选择联系人
  ↓
粘贴对方消息
  ↓
点击“幽默 / 真诚 / 暧昧”等提示词
  ↓
发送
  ↓
在聊天模拟区检查是否自然
  ↓
复制回复
```

# 11. 组件拆分建议

当前仓库中的主界面与下述结构**实质对齐**（命名以文件为准；细粒度子组件可继续内联或下拆）：

```text
page.tsx
├── ContactSidebar
│   └── 搜索、新建、两行列表项
├── ChatWorkspace
│   ├── ContactHeader（单行 + hover 完整信息）
│   ├── ConversationArea（双栏：左侧 SuggestionCard 列表，右侧聊天气泡预览）
```

可演进方向（与当前代码无强制一一对应）：独立 `ContactInfoTooltip`、可复用的 `EmptyState` 子组件、将 `PromptChips` 抽为无状态列表等。

# 12. 数据结构草案

以下与 `src/lib/types.ts` **一致**；阶段四再引入富输入时，再扩展 `ComposerDraft` 等。

## 12.1 ContactSkill 与 Contact 基类

```ts
type ContactSkill = {
  manual_tags: string[]
  distilled_traits: DistilledTrait[]
  notes?: string
  updated_at: string
}

type DistilledTrait = { key: string; value: string; confidence?: number }

type Contact = {
  id: string
  name: string
  skill: ContactSkill
}
```

## 12.2 WorkspaceContact

当前主界面使用的联系人（在 `Contact` 上增加列表与 Header 展示字段）：

```ts
type WorkspaceContact = Contact & {
  relation: string
  attributeDefinition: string
  lastActive: string
  summary: string
}
```

## 12.3 ComposerDraft

已正式取消单独的 `topic` 字段。提示词 chip 只通过修改 `expectation` 字符串体现，不另存数组或附件。

```ts
type ComposerDraft = {
  incomingMessage: string
  expectation: string
}
```

## 12.4 ContactWorkspaceState

`contactId` 不作为嵌套字段存储：由页面级 `Record<contactId, ContactWorkspaceState>` 的键表示当前归属。

```ts
type ContactWorkspaceState = {
  draft: ComposerDraft
  suggestions: Suggestion[]
  selectedSuggestionId?: string
  error: string | null
}
```

## 12.5 ComposerAttachment（阶段四预留，当前未使用）

```ts
type ComposerAttachment = {
  id: string
  type: 'link' | 'image' | 'file' | 'sticker'
  title: string
  source?: string
  metadata?: Record<string, unknown>
}
```

## 12.6 Suggestion

```ts
type Suggestion = {
  id: string
  style: string
  text: string
}
```

## 12.7 AiSuggestionRequest、Intent、Message（节选）

```ts
type Intent = { id: string; label: string; description: string; custom?: boolean }

type Message = {
  id: string
  sender: 'peer' | 'self'
  text: string
  timestamp?: string
}

type AiSuggestionRequest = {
  model: string
  contact: WorkspaceContact
  draft: ComposerDraft
  intent: Intent
  recentMessages: Message[]
  workspaceContext: { source: 'manual-paste'; generatedAt: string }
}
```

# 13. 状态矩阵

| 场景 | 左侧联系人列表 | 右侧工作台 | 发送按钮 |
| --- | --- | --- | --- |
| 无联系人 | 展示创建入口 | 展示引导空态 | 禁用 |
| 未选择联系人 | 可选择联系人 | 提示选择联系人 | 禁用 |
| 已选择联系人但无消息 | 高亮联系人 | 候选区与模拟区展示输入引导 | 禁用 |
| 有消息且选 Mock | 正常 | 可发送；模拟区随输入更新 | 可用（无 API Key 要求） |
| 有消息且选 OpenAI-compatible 但未在桌面端保存有效配置 | 正常 | 可点发送，请求失败后展示错误并引导「配置模型」 | 可用（点击后失败） |
| 有消息且选 OpenAI-compatible 且配置有效 / 或 Mock | 正常 | 可发送，模拟区展示联系人消息与预览 | 可用 |
| AI 请求中 | 正常 | 候选区展示生成中，模拟区保留上下文 | loading |
| AI 失败 | 正常 | 展示错误与重试 | 可重试 |
| AI 成功 | 正常 | 展示 2-3 条建议，模拟区预览回复 | 可再次发送 |
| 切换联系人 | 高亮新联系人 | 恢复该联系人独立状态 | 根据该联系人输入状态决定 |

# 14. 视觉方向

## 14.1 整体风格

- 桌面工具感
- 干净、轻量、低干扰
- 左侧偏导航，右侧偏工作区
- 候选回复突出可复制性
- 聊天模拟区突出“放进真实聊天里的效果”

## 14.2 布局建议

- 左侧宽度：240-280px
- 右侧自适应
- 顶部联系人 Header 高度：40-48px
- 中间主体双栏自适应撑满
- 聊天模拟区宽度：280-360px
- 底部输入区根据输入内容自动增高，但需要设置最大高度
- 两个输入框默认高度都应紧凑，避免占用过多垂直空间

## 14.3 颜色建议

- 背景：浅灰或深色模式均可支持
- 当前联系人：使用轻微高亮背景
- 发送按钮：主品牌色
- 禁用未来功能：低对比度 + tooltip
- AI 建议卡片：白底 / 深色卡片 + 轻阴影
- 聊天模拟区：接近真实聊天气泡，但保持轻量工具感
